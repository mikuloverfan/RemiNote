# 结构化拓扑匹配方案

基于对用户需求（"至少齐平手写输入法"）的重新思考，提出**拓扑结构 + 笔画骨架分析**方案。

## 核心思想

手写输入法用深度学习模型做序列分类。但我们换一个角度：**汉字有很强的拓扑结构**——"口"一定包含一个闭环，"十"一定有交叉点，"三"一定是三条横线。这些结构特征在手写中**高度稳定**，不需要神经网络也能提取。

这与 CAPTCHA 识别 利用的视觉特征原理相同，只是我们反过来用——用结构特征**匹配**而非**识别**。

## 特征体系

### ① 拓扑特征（8 维度）

| 特征 | 提取方式 | 示例 |
|------|----------|------|
| **闭环数** | 从渲染骨架图检测连通区域中的空洞 | 口=1, 田=4, 日=1, 目=1, 八=0, 大=0 |
| **交叉点数** | 笔画交点检测 | 十=1, 米=4, 本=2, 口=0 |
| **端点数** | 笔画端点计数 | 一=2, 二=4, 三=6 |
| **连通分量** | 分离的笔画组数 | 二=2, 三=3, 心=1 |
| **外包围盒形状** | w/h 比率离散化 | 口≈1, 日≈1.5, 目≈2 |
| **像素密度分布** | 4×4 网格灰度直方图 | 对称性特征 |
| **水平投影波峰** | 每行黑色像素数 | "三"=3个波峰，"川"=3个波峰 |
| **垂直投影波峰** | 每列黑色像素数 | "川"=3个波峰，"三"=1个波峰 |

### ② 笔画级特征（每笔 6 维）

| 特征 | 说明 |
|------|------|
| 方向量化 | 8 方向码，每笔压缩为等长序列 |
| 弯曲度 | 弧长/弦长比 |
| 相对位置 | 笔画重心在字内的归一化位置 (x,y) |
| 占空比 | 笔画覆盖的字形比例 |
| 起止方向 | 起笔和收笔的 8 方向量化 |

## 匹配流程

```
笔画输入（用户手写）
    │
    ▼
特征提取
    ├─ 渲染到 256×256 骨架图 → 连通域分析 → 闭环检测
    ├─ 笔画交叉点检测
    ├─ 端点、投影、密度分布
    └─ 笔画级方向序列
    │
    ▼
四级过滤
    ├─ T1: 闭环数 (精确匹配)            → 95% 候选淘汰
    ├─ T2: 笔画数 ±1                    → 再淘汰 50%
    ├─ T3: 交叉点数 ±1 + 连通分量数      → 再淘汰 60%
    ├─ T4: 投影波峰数 ±1                 → 再淘汰 40%
    │                                     剩余候选通常 <10 个
    ▼
精确匹配
    └─ 多级 NCC (256px → 128px → 64px)
        + 骨架 Hausdorff 距离
        → 最高分候选输出
```

## 为什么这样可行

仅用 **T1(闭环数) + T2(笔画数)** 就可以区分的常见字：

| 闭环数 | 笔画数 | 候选范围 |
|--------|--------|----------|
| 0 | 1-3 | 一、二、三、十、人、八、大、小... ~30字 |
| 1 | 5-6 | 口、日、目、田、白、四... ~40字 |
| 2 | 6-8 | 目+八=具? 不，闭环位置区分 日+一=旦... |
| 3+ | 8+ | 田+口=... 但笔画数不同 |

加上 T3 (交叉点) 和 T4 (投影)，绝大多数情况下候选降至 **3-5 个**，此时 256px NCC 足够做出精确判断。

## 预期效果

| 指标 | 值 |
|------|-----|
| 准确率 | 80-90%（工整手写） |
| 速度 | ~5-10ms |
| 依赖 | 0（纯数学运算） |
| 内存 | ~500KB 特征库 |
| main.js 增量 | ~30KB |

## 对比

| | 几何特征匹配（已否决） | 结构化拓扑匹配（新方案） |
|---|---|---|
| 特征基础 | 低级数值向量 | 高级结构语义 |
| 对书写风格敏感度 | 高 | 低 |
| 区分度 | 一般 | 强（闭环、交叉点是语义级特征） |
| 潦草适应性 | 差 | 较好（闭环结构在潦草时仍存在） |
| 实现复杂度 | 低 | 中 |

## 实施步骤

1. 创建 `SkeletonExtractor.ts` — 笔画渲染 → 骨架化 → 特征提取
2. 创建 `FeatureDB.ts` — 基于字体渲染生成 1600 字特征库（闭环数、交叉数、投影等）
3. 创建 `StructureMatcher.ts` — 四级过滤 + 精确匹配
4. 集成到 `RedrawOrchestrator.ts` — 替换 PaddleOCR 路径
5. 清理废弃代码（paddleocr、onnxruntime-web 等）
